## 3. Format Evidence
- container/header: decompressed fixture starts with `Sony PS2 Memory Card Format `; backend PS2 reader reports page size 512, pages per cluster 2, cluster size 1024, allocatable cluster offset 41, allocatable cluster end 8135.
- slot layout: root directory contains `BASLUS-21050`, `BASLUS-21087`, and `BADATA-SYSTEM`; `BADATA-SYSTEM` is system configuration and not a user save.
- game directory: `BASLUS-21087` matches the PS2 directory product-code pattern and derives product code `BASLUS-21087`.
- directory contents observed: only `icon.sys` and `game.icn` are present for the Mortal Kombat entry in the available fixture.
- title proof: `icon.sys` has magic `PS2D`, title offset `0x000e`, title line one `Mortal Kombat`, title line two `Shaolin Monks`, and normal icon file `game.icn`.
- checksum/crc: no gameplay checksum or repair rule was proven from the available bytes.
- mirrored data: no mirrored gameplay data was observed.
- parser/doc sources: existing backend PS2 memory-card parser, local fixture inspection, PSXDataCenter serial/size listing, GameFAQs save listing descriptions, public PS2 `icon.sys` documentation.
- confidence: high for PS2 card/directory/title identification; low for gameplay fields because no gameplay data file was present in the available raw sample.

## 6. Required Backend Logic

- new parser/editor needed: yes. Implemented as `modules/mortal-kombat-shaolin-monks.rsmodule.zip`, a sandboxed read-only PS2 logical-save module.
- checksum repair required: unknown. Must be proven from a real gameplay-bearing save before writes.
- mirrored-write required: unknown.
- current backend support: when `psLogicalKey` is supplied, the backend sends the selected PS2 logical save ZIP to module `parse`, `readCheats`, and `applyCheats`. This module uses that path and does not parse the full 8 MiB card.
